# 専門用語スキマ学習アプリ 実現計画（MVP中心）

## 1. 目的とゴール

本ドキュメントは、READMEに定義された「通知起点で専門用語を短時間学習できるAndroidアプリ」を、
**MVPとして確実にリリースするための実行計画**をまとめたものです。

### 1.1 プロダクトゴール
- 忙しいユーザーが「1回30秒〜2分」で学習できる体験を成立させる
- 通知・ウィジェットを起点に学習を日常導線へ組み込む
- 忘れやすい単語の再出題で、少ない時間でも定着率を高める

### 1.2 MVP完了条件（Definition of Done）
以下を実機Androidで確認できたらMVP完了とする。
- ジャンル/難易度を設定し、その条件で出題される
- 通知から3択クイズに回答できる
- 回答結果（正誤）が保存される
- 忘却曲線ベースで次回出題時刻が更新される
- ウィジェットで単語を表示できる
- 出題一覧および正解/不正解分類画面で履歴を見られる

---

## 2. スコープ管理

### 2.1 MVPスコープ（実装対象）
- Androidアプリ（日本語UI）
- 初期ジャンル: IT・開発系
- 日本語話者向け英語専門用語学習
- 必須機能（README記載のMVP機能）

### 2.2 非スコープ（MVPでは実装しない）
- ランキング、一斉テスト、SNS機能
- AI苦手分析（ルールベースの復習優先度のみ実装）
- 多言語同時対応

### 2.3 変更管理ルール
- 新機能追加は「MVPゴールへの寄与」を満たす場合のみ
- スプリント中の仕様変更は、影響範囲（UI/DB/通知）を明記して承認後に反映

---

## 3. システム設計方針

## 3.1 全体アーキテクチャ（推奨）
- クライアント完結型（初期はオフライン優先）
- Android: Kotlin + Jetpack Compose
- 構成: `Presentation / Domain / Data` の3層
- 永続化: Room（単語・学習履歴・出題スケジュール）
- バックグラウンド: WorkManager（通知スケジュール）

## 3.2 主要コンポーネント
1. **学習設定モジュール**
   - ジャンル選択、難易度選択、通知時刻設定
2. **出題エンジン**
   - 出題候補抽出、誤答選択肢生成、次回出題日時計算
3. **通知/クイズ導線**
   - 通知表示、通知アクションまたは即時回答画面遷移
4. **履歴/可視化モジュール**
   - 出題一覧、正解/不正解タブ
5. **ウィジェットモジュール**
   - 単語表示、タップでアプリ遷移

## 3.3 データモデル（最低限）
- `term_master`
  - id, term, reading_katakana, meaning_ja, genre, difficulty, example_en, example_ja
- `term_choices`
  - id, term_id, wrong_choice_1, wrong_choice_2
- `learning_state`
  - term_id, correct_count, incorrect_count, last_answered_at, next_due_at, stability_level
- `answer_history`
  - id, term_id, answered_at, is_correct, response_time_ms
- `app_settings`
  - selected_genres, selected_difficulty, notification_time, daily_limit

## 3.4 再出題ロジック（初期版）
- 初回正解: +1日
- 2回連続正解: +3日
- 3回連続正解: +7日
- 不正解: +10分〜当日内で再出題
- `next_due_at <= now` の単語を優先し、未出題語を一定割合混ぜる（例: 70%復習 / 30%新規）

## 4. 実装ロードマップ

## Phase 0: 要件確定・プロトタイプ（1週間）
- 画面フロー確定（設定→通知→回答→結果→履歴）
- ワイヤーフレーム作成
- 単語データCSVフォーマット決定

**成果物**
- 画面遷移図
- データ定義（ER図簡易版）
- MVP受け入れ基準チェックリスト

## Phase 1: 基盤実装（1〜2週間）
- プロジェクト雛形作成（Compose, DI, Room）
- マスターデータ投入処理（CSV import）
- 設定画面（ジャンル/難易度/通知時刻）

**完了判定**
- 設定保存・再読込が動作
- 任意単語をDBから取得して表示可能

## Phase 2: 出題と通知導線（2週間）
- 出題エンジン初版
- WorkManagerによる通知定期実行
- 通知タップでクイズ画面表示
- 3択回答と正誤判定

**完了判定**
- 実機で通知受信→回答→結果表示まで通る

## Phase 3: 復習最適化と履歴（1〜2週間）
- 忘却曲線ベース再出題反映
- 学習履歴保存
- 出題一覧、正解/不正解分類表示
- ウィジェット表示

**完了判定**
- 正誤履歴が保存され、次回出題に反映される
- ウィジェットから学習導線へ遷移できる

## Phase 4: リリース準備（1週間）
- クラッシュ/ANR重点確認
- バッテリー最適化影響の確認
- 利用規約・プライバシーポリシー整備（最低限）
- 内部テスト配布

**完了判定**
- MVP完了条件を満たし、内部テストで重大バグなし

## 5. 体制と役割（最小構成）
- PM/企画: 1名
- Android開発: 1〜2名
- デザイン: 0.5名（兼任可）
- 用語データ作成: 1名（兼任可）

### 5.1 運用ルール
- 週1回の進捗レビュー
- スプリント終了時にMVP受け入れ基準で評価
- 仕様変更はIssue化して履歴を残す

---

## 6. リスクと対策

1. **通知がOS制約で遅延/欠落する**
   - 対策: WorkManager + exact alarm要否評価、通知許可導線の明示
2. **単語データ品質不足（誤答が不自然）**
   - 対策: データレビュー観点を定義（意味重複、難易度ズレ）
3. **復習ロジックが難しすぎてチューニング不能**
   - 対策: 段階導入（ルールベース→将来SM-2等）
4. **機能増加によるMVP遅延**
   - 対策: 非スコープ厳守、Phase 4前の新規機能凍結

---

## 7. KPI（MVP検証用）

- D1継続率
- 1日あたり回答数（平均）
- 通知開封率
- 正答率の推移（初回週→4週目）
- 再出題対象の正答改善率

### 7.1 計測方針
- MVP段階ではローカルログ + 簡易分析で可
- 本格分析基盤は正式リリース後に追加
